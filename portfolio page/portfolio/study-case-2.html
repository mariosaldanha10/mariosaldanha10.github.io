The uses cases of Amazon S3 is that this resource can be used in Data Lake and big Data Analytics as well as Backup and restoration. Steps to configure Auto Scaling Group with Auto Load Balancer


Step 1 – Create a VPC A VPC was created on the Networking & Content Delivery Services dashboard using the option of create VPC and more which was named as digitech-webserver-vpc. The IPv4 CIDR block chosen was 10.1.0.0/16. Three Number of Availability Zones was chosen in order to provide High Availability. Two numbers of public subnets were chosen along with two numbers of private subnets. No NAT gateways was chosen. On VPC endpoints the S3 Gateway was chosen and on DNS options the DNS hostnames and DNS resolution were enabled.

Step 2 – Launch Template The Launch Template name was defined by “digitech-webservers-template” and the Template version description was defined by “A webserver for my website”. The Auto Scaling Guidance was selected in order to help me out to configure an ASG after all. The AMI required was set up by Amazon Linux 2023 AMI and the Instance Type selected was t2.micro (free tier eligible), my own Key Pair was used named by Year4-Cloud-Class-key. On the Network Settings The Security Group named by “digitech-webserver-01” was selected and an Auto-assign Public IP was set and also Delete on Termination was set. In order to create an ASG is necessary to Launch Template first where it will have the settings for the instances. These settings include AMI (Amazon Machine Image), the type of instance, the key pair and security group. The name of the Template was defined by “digitech-webservers-template”

Step 3 – Security Group This security group is named by “digitech-webserver-01” where the inbound type rules is configured by HTTP and SSH where the source IP is set by Anywhere-IPv4 0.0.0.0/0 and the outbound type rules is configured by All Traffic where the destination is set by Anywhere-IPv4 0.0.0.0/0.

Step 4 – Load Balancers This load balancer is named by “digitech-webserver-01” and in the configuration is using the “digitech-webserver-vpc” and mapping three Availability Zones such as us-east-1a (use1-az1), us-east-1b (use1-az2), us-east-1c (use1-az4) and using all the public subnets such as (digitech-webserver-subnet-public1-us-east-1a), (digitech-webserver-subnet-public2-us-east-1b) and (digitech-webserver-subnet-public3-us-east-1c). Finally, the security group is set by “digitech-webserver-01”.

Step 5 – Target Groups The target type chosen was instances and the target group name was defined by digitech-webserver-01. The VPC selected was “digitech-webserver-vpc” and all the instances I have created before to make the task 2 were selected.

Step 6 – Auto Scaling Group Step 1 – Chosen launch template: The ASG name was defined by “digitech-webservers-01” and the Launch Template chosen was “digitech-webservers-template
Step 2 – Chose instance launch options: A VPC was chosen as “digitech-webserver-vpc” and the public subnets such as (digitech-webserver-subnet-public1-us-east-1a), (digitech-webserver-subnet-public2-us-east-1b) and(digitech-webserver-subnet-public3-us-east-1c) were chosen.
Step 3 – Configure advanced options: An existing load balancer target group such as “digitech-webserver-01 | HTTP” was chosen.
Step 4 – Configure group size and scaling: in this step all the configuration was left as it is. Step 5 – Add notifications: no notifications was added. Step 6 – Add Tags: no tags were added and the ASG was created.

Subject: Implementation of Auto Scaling Group with an Application Load Balancer for DigiTech Website


The objective of this architecture proof of concept is to demonstrate the deployment of a powerful and scalable infrastructure for hosting DigiTech’s website. Along with AWS providing its services of Auto Scaling Group and Application Load Balancer and how they work together and its advantages. Details of the Implementation: Virtual Private Cloud (VPC) is where you can manage and operate all your data within a secluded virtual network defined by you. There is the advantage of manage and use scalable infrastructure of AWS. Auto Scaling Group (ASG) is when you define how many instances you want to set and you define a policy, for example, if you define that the ASG can have only up to 4 instances, this is what is going to be the rules. Auto Scaling Group will make sure that your instances never go beyond that and also, if one of the instances becomes unhealthy, another instance takes over. Application Load Balancer (ALB) is to make sure that all the traffic coming across the instances are well and organized distributed. This feature has the power to balance the traffic to make sure and prevent the instances for being overwhelmed. The advantages of having an integration of Auto Scaling Group and Application Load Balancer working together is that they provide powerful benefits. One of these benefits that are key in this resource is scalability. If there is a lot of traffic coming across the website then the ASG can handle it increasing the power and on the other hand you have ALB where handles and distribute very well the amount of traffic. Another advantage of them working together is the high availability they provide. For example, if there is a problem in one of the instances and this instance becomes unavailable, AGS automatically detects and replaces another instance straightaway. Not to mention about the optimization related to costs where if the traffic is low and no instances are being used, then AGS reduces the number of instances and consequently this affects in low costs

Concepts of Auto Scaling and Elastic Load Balancing


Imagine that you have a website for your Retailing Shop and you wish to implement one of the AWS services. The services you wish to implement are to improve scalability and traffic management. One of the features that AWS provides is Auto Scaling which is responsible to manage all the number of existing hosts and the number of visitors accessing your website. Bare in mind that a relatively high number of people are visiting your website at the same time. Consequently, your website needs more power due to a large number of accesses at the same time. Then the job of Auto Scaling is to add more computers automatically. However, if no customers are around, then the job of Auto Scaling is to remove unnecessary computers to save costs. When working together with Elastic Load Balancing (ELB), the job of this service is to distribute all the traffic from the website to every other instances. This is to make sure that not a single instance will be overwhelmed with traffic. Furthermore, this is to make sure that all instances are as well running smoothly. Also, when implementing Auto Scaling and Elastic Load Balancing to your website, this provides Higher Availability and Redundancy. When having High Availability and Redundancy, this is the key to guarantee that your website is always up and running 24/7 no matter what happen. Also, AWS Services provides various different Availability Zones for your computers in the same region. The advantage of this is that if one AZ has an issue, Auto Scaling works to shift the workload to another healthy AZ, so your website is always up and running. Additionally, the Redundancy resource makes copies of your instances acting as a backup for different AZs, making sure nothing goes wrong with the availability of your website.
